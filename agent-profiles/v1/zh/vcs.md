# VCS（中文对照）

Source of truth:
[../en/vcs.md](../en/vcs.md)

本文档是英文版的中文对照参考。
若中英文存在差异，以英文版为准。

## 何时阅读

当涉及 commit、分支、合并、rebase、创建 PR/MR 或对版本控制状态做任何变更时。

## 何时不必阅读

当变更仅是本地编辑、尚未进入版本控制时。

## 默认值

把版本控制状态视为用户所有的历史记录。
未经明确指示，不要修改共享或已发布的状态。

## 硬性默认

- 永远不要自动 commit；仅在用户明确要求时执行
- 永远不要自动 push；仅在用户明确要求时执行
- 永远不要跳过 hook（`--no-verify`、`--no-gpg-sign` 等），除非用户明确授权
- 永远不要 force-push 到受保护或 main 分支；如果用户要求，应先警告
- 优先创建新 commit，而不是 amend 已发布的 commit
- 按文件名显式 stage，避免 `git add -A` / `git add .`，以免误带 secret、生成物或无关变更

## 提交粒度

- 一次 commit，一件独立的事
- 在可行时，把重构与行为变化分开
- 如果一个任务两者都涉及，先 commit 重构，再 commit 行为变化

## Commit 信息

- 沿用项目现有风格；动笔前先看最近 log
- 当仅写"做了什么"不足以说明问题时，在 body 中说明"为什么"
- 项目没有约定时，不要凭空发明一种；保持简短描述即可

## 冲突与恢复

- 在执行破坏性操作（`git reset --hard`、`git checkout --`、`git clean -f`、删除分支）前先调查
- 解决合并冲突，不要把冲突的一边直接丢掉以图省事
- lock 文件、陌生分支、未跟踪文件可能是用户正在进行的工作，删除前先确认

## Pull / Merge Request

- 仅在用户要求时创建
- 标题简短，详情放 body
- 汇总从 base 分支分叉以来的完整 diff，而不只是最新 commit

## 并发

- 不要并行运行可能竞争 git index 锁的命令

## 汇报

执行任何 commit、push 或 PR 动作后，应汇报：

- 执行了什么动作
- 可见的副作用（commit SHA、分支状态、PR URL）
- 是否还有需要用户确认的事项
